## HW 问题收集

列举在HW 1、HW1.1过程里，你所遇到的2~3个通过自己学习已经解决的问题，和2~3个尚未解决的问题与挑战

### 已解决

1. 一开始不清楚 `Sudoku` 和 `Game` 的职责怎么分
   1. **上下文**：最初我把校验、历史、输入处理混在一起，导致代码边界不清晰，改一点就牵一片。
   2. **解决手段**：复盘作业要求后重新划分职责：`Sudoku` 负责盘面与规则（`guess/getInvalidCells/toJSON`），`Game` 负责流程与历史（`guess/undo/redo/canUndo/canRedo`）。

2. 不知道怎么把领域对象“真正接到 Svelte 页面里”
   1. **上下文**：之前页面主要还是直接操作 store 里的数组，领域对象在测试里可用，但 UI 流程没有真正经过它。
   2. **解决手段**：采用 store adapter 思路，在 `stores/grid.js` 内部持有 `Game`，由适配层统一对外暴露状态和命令（`set/applyHint/undo/redo`），组件只负责渲染和调用。

3. 不理解为什么改了对象内部字段，界面有时不刷新
   1. **上下文**：直接 mutate 嵌套对象/数组时，出现过“数据变了但页面没按预期更新”的情况。
   2. **解决手段**：改为统一通过 store 发布新状态（`update` + 新快照），让 UI 全部通过 `$store` 消费，避免在组件层直接改内部对象。

### 未解决

1. 本地构建链路仍有依赖兼容问题
   1. **上下文**：`npm run build` 报 `node.getIterator is not a function`，定位在 `svelte-preprocess + postcss` 链路。
   2. **尝试解决手段**：先保证作业核心功能和测试通过（`npm test` 已通过），后续再专门对齐相关依赖版本并回归验证。

2. 适配层职责仍偏重，后续还想继续简化
   1. **上下文**：当前方案可用，但 `stores/grid.js` 里状态映射代码较多，可维护性还有优化空间。
   2. **尝试解决手段**：已在 `DESIGN.md` 记录 trade-off；计划继续抽取更稳定的快照转换与状态导出接口，减少重复映射逻辑。

3. 对“领域层可扩展性”的把握还不够有信心
   1. **上下文**：目前已支持接入与 Undo/Redo，但如果后续加更复杂功能（如更丰富提示策略），不确定现有抽象是否仍最优。
   2. **尝试解决手段**：通过补测试和小步重构来验证设计稳定性，优先保持 `Game` 作为唯一业务入口，避免逻辑回流到组件层。
